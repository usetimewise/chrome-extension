# Live-проверки Chrome extension

Для live-проверок расширения используй Chromium-Gost, а не стандартный Chrome DevTools MCP-браузер.

Причины:
* браузер, которым управляет `chrome-devtools` MCP, запускается с `--disable-extensions`, поэтому расширение там не загрузится;
* Google Chrome на этой машине игнорирует `--load-extension` для unpacked extension и пишет в логах `--load-extension is not allowed in Google Chrome, ignoring`;
* Chromium-Gost корректно загружает unpacked extension из `dist` и доступен по `/Applications/Chromium-Gost.app/Contents/MacOS/Chromium-Gost`.

Рабочий запуск для live smoke:

```bash
/Applications/Chromium-Gost.app/Contents/MacOS/Chromium-Gost \
  --user-data-dir=/tmp/zalipoff-extension-live-test \
  --load-extension=/Users/ikochergin/work/time-wise-tracker-chrome/extension/dist \
  --disable-extensions-except=/Users/ikochergin/work/time-wise-tracker-chrome/extension/dist \
  --remote-debugging-port=9227 \
  --no-first-run \
  --no-default-browser-check \
  about:blank
```

Важно:
* перед запуском live smoke сначала выполни `npm run build`, потому что проверяется именно `dist`;
* запускай команду с escalated permissions: это GUI-приложение и локальный DevTools порт;
* дождись строки `DevTools listening on ws://127.0.0.1:9227/...` в выводе процесса, первые `curl` к порту могут упасть, если браузер еще поднимается;
* extension id ищи в `/tmp/zalipoff-extension-live-test/Default/Secure Preferences`, а не только в `Preferences`; для этого проекта при загрузке `dist` id обычно `ooocnckfllceadgfjbihhjjedjdhejfm`;
* CDP endpoint `/json/new?...` в Chromium-Gost требует `PUT`, `GET` возвращает `405`;
* popup открывай как `chrome-extension://<extension-id>/popup.html`;
* в popup кнопка настроек имеет selector `.settings-icon-button` и `aria-label="Открыть настройки"`;
* после проверки закрой тестовый браузер через `pkill -f zalipoff-extension-live-test`, чтобы следующий smoke не цеплялся к старому профилю или вкладкам.

# Основные принципы

## 1. Production-ready подход

Пиши код так, будто он сразу попадет в production.

Код должен быть:

- читаемым;
- типобезопасным;
- устойчивым к ошибкам;
- простым для изменения;
- минимально связанным с другими частями приложения;
- без временных решений, заглушек и неиспользуемого кода;
- без `any`, если только это не абсолютно неизбежно;
- без подавления ошибок через `// @ts-ignore`, `eslint-disable` без объяснения.

Если есть неоднозначность — выбери наиболее безопасное и простое решение. Не усложняй архитектуру без необходимости.

# TypeScript

## 5. Строгая типизация

Используй TypeScript как инструмент проектирования.

Правила:

* не используй `any`;
* предпочитай `unknown` вместо `any`, если тип действительно неизвестен;
* явно типизируй публичные API функций, хуков и компонентов;
* используй discriminated unions для состояний;
* не дублируй типы, если их можно вывести;
* используй `type` или `interface` последовательно согласно стилю проекта;
* не расширяй типы без необходимости;
* избегай чрезмерно сложных generic-типов.

Пример хорошей модели состояния:

```ts
type RequestState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; error: string };
```

Не используй nullable-кашу вида:

```ts
{
  data: T | null;
  isLoading: boolean;
  error: string | null;
}
```

если есть риск несогласованных состояний.

---

## 6. Типы данных с backend

Не доверяй данным с сервера.

Если данные приходят извне:

* валидируй их при необходимости;
* учитывай отсутствие полей;
* не делай небезопасных assertion через `as`;
* отделяй DTO от доменной модели, если структура API неудобна для UI.

Пример:

```ts
type UserDto = {
  id: string;
  first_name: string;
  last_name: string;
};

type User = {
  id: string;
  fullName: string;
};

function mapUserDto(dto: UserDto): User {
  return {
    id: dto.id,
    fullName: `${dto.first_name} ${dto.last_name}`,
  };
}
```

# React

## 7. Компоненты

Пиши компоненты как чистые, предсказуемые функции.

Правила:

* компонент должен иметь одну основную ответственность;
* props должны быть явно типизированы;
* избегай больших компонентов;
* выноси повторяющуюся логику в хуки;
* не храни derived state без необходимости;
* не используй `useEffect` для того, что можно вычислить во время рендера;
* не злоупотребляй `useMemo` и `useCallback`;
* не делай premature optimization;
* не создавай функции и объекты вне компонента только ради «оптимизации», если это ухудшает читаемость.

Плохой подход:

```tsx
const [fullName, setFullName] = useState('');

useEffect(() => {
  setFullName(`${firstName} ${lastName}`);
}, [firstName, lastName]);
```

Хороший подход:

```tsx
const fullName = `${firstName} ${lastName}`;
```

---

## 8. Hooks

Кастомные хуки должны инкапсулировать сценарную или повторяемую логику.

Правила:

* имя начинается с `use`;
* не смешивай в одном хуке слишком много обязанностей;
* возвращай стабильный и понятный API;
* не прячь побочные эффекты там, где вызывающий код должен знать о них;
* не делай хук универсальным раньше времени.

Пример:

```ts
function useUserProfile(userId: string) {
  return useQuery({
    queryKey: ['user-profile', userId],
    queryFn: () => getUserProfile(userId),
    enabled: Boolean(userId),
  });
}
```

---

## 9. Работа с состоянием

Выбирай минимально достаточный уровень состояния.

Приоритет:

1. локальный state компонента;
2. state в родителе;
3. context;
4. серверное состояние через TanStack Query или аналог;
5. глобальный store.

Не клади всё подряд в глобальное состояние.

Глобальное состояние допустимо для:

* авторизации;
* пользовательских настроек;
* feature flags;
* кросс-страничных UI-состояний;
* данных, которые реально нужны во многих частях приложения.

Не используй глобальный store как кэш API, если в проекте есть TanStack Query.

# API и side effects

## 10. Работа с API

Изолируй API-запросы от UI.

Правила:

* не вызывай `fetch`/`axios` прямо внутри JSX-компонентов, если в проекте есть слой API;
* централизуй base URL, headers, auth, обработку ошибок;
* типизируй вход и выход;
* обрабатывай loading, empty, error и success состояния;
* не игнорируй race conditions;
* используй abort/cancellation там, где это необходимо.

Пример структуры:

```txt
features/user/
  api/
    getUser.ts
  model/
    types.ts
  ui/
    UserCard.tsx
```

---

## 11. Ошибки

Обрабатывай ошибки явно.

UI должен иметь состояния:

* loading;
* error;
* empty;
* success.

Ошибки должны быть полезными для пользователя, но без утечки технических деталей.

Для разработчика оставляй достаточно информации через логирование, если в проекте есть инфраструктура логов.

Не пиши:

```ts
catch (e) {
  console.log(e);
}
```

Лучше:

```ts
catch (error) {
  throw normalizeApiError(error);
}
```

## 12. Формы

# Качество кода

## 18. Именование

Имена должны объяснять назначение.

Хорошо:

```ts
const isSubmitDisabled = isLoading || !formState.isValid;
```

Плохо:

```ts
const flag = loading || !valid;
```

Правила:

* boolean-переменные начинаются с `is`, `has`, `can`, `should`;
* обработчики событий — `handleSubmit`, `handleChange`, `handleClose`;
* компоненты — PascalCase;
* функции и переменные — camelCase;
* константы — UPPER_SNAKE_CASE только для настоящих констант уровня модуля;
* файлы называй согласно стилю проекта.

---

## 19. Импорты

Следуй текущему стилю импортов.

Обычно порядок такой:

1. внешние библиотеки;
2. app-level imports;
3. pages/features/entities/shared;
4. относительные импорты;
5. стили.

Удаляй неиспользуемые импорты.

Не создавай циклические зависимости.

---

## 20. Комментарии

Комментарии нужны только там, где код не может сам объяснить причину.

Не комментируй очевидное.

Плохо:

```ts
// Increment counter by one
setCount(count + 1);
```

Хорошо:

```ts
// Backend expects price in cents to avoid floating-point rounding issues.
const priceInCents = Math.round(price * 100);
```

# Главный принцип

Пиши не просто рабочий код, а код, который другой senior-разработчик сможет без проблем читать, ревьюить, тестировать и развивать.
